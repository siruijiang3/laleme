# LaLeMe Roadmap

LaLeMe 的路线图以验证真实需求为优先级。先做一个能被普通用户使用的 Web/PWA MVP，再逐步增强数据质量、场景适配和平台能力。

## MVP：能用的开放厕所地图

目标：让用户可以在手机浏览器里找到厕所、贡献厕所、更新厕所状态。

当前状态：已有生产标准 Web/PWA 原型，使用 Next.js + TypeScript、MapLibre GL JS 和 Hosted Supabase/Postgres。生产环境不保留 mock fallback、seed fixture、占位地图或 local-only 写入。

包含：

- Web/PWA 地图首页
- 附近厕所点位展示
- 新增厕所点位
- 厕所详情页
- 开放状态更新
- 是否有厕纸更新
- 清洁度更新
- 无障碍信息
- 评分和评论
- 简单厕纸求助
- 自有数据库保存数据
- OpenStreetMap `amenity=toilets` Geofabrik extracts 导入作为开放数据底座

不包含：

- 原生 iOS App
- 复杂登录系统
- 复杂审核后台
- 第三方厕所 POI 导入
- 商业地图或闭源 POI 导入
- 实时聊天
- 商业化功能
- 推荐系统

MVP 完成标准：

- 用户打开网页后能看到地图。
- 用户能新增一个厕所。
- 其他用户能看到这个厕所。
- 用户能更新厕所状态。
- 用户能查看详情、评分和评论。
- 用户能发起简单求助。
- 数据保存在 LaLeMe 自己的数据库中。

当前原型已加入 Supabase/Postgres schema、按地图范围读取 API、生产配置检查和服务端写入路径。真实账号系统仍属于后续工作。

当前 Web 原型已改为附近厕所优先，不再把校园、商场、地铁站作为首页主切换。厕所详情分享链接和最小 admin 举报处理页已加入；admin 页面只解决公开试用前的基础运营闭环，不等于完整审核后台。

当前数据层已加入 Geofabrik OSM 同步脚本，用于按国家/区域 extracts 导入 `amenity=toilets`。这只是开放数据底座，不替代用户贡献和实时状态确认。

生产部署的数据一致性要求是 Hosted Supabase 单一真源。生产环境不使用 mock fallback，也不保留本地-only 写入结果。

## 第二阶段：提高数据可信度

目标：让用户知道哪些厕所信息可信、哪些需要再次确认。

可能包含：

- 最近确认时间
- 重复点位合并
- OSM 导入点位与用户贡献点位合并
- 错误信息举报
- 更完整的管理页面
- 贡献记录
- 轻量信誉机制
- 求助过期和已解决状态

完成标准：

- 明显错误数据可以被处理。
- 重复点位不会长期堆积。
- 用户能判断厕所信息是否新鲜。
- 求助信息不会永久残留。

## 未来阶段：更多真实场景

目标：让 LaLeMe 适配更多复杂空间和使用场景。

可能包含：

- 商场楼层信息
- 地铁站和机场区域说明
- 景区、公园、办公园区适配
- 营业时间
- 入口描述
- 图片上传
- 更多筛选条件
- 多语言支持
- 原生 iOS App 评估

未来阶段不应牺牲核心原则：简单、开放、可验证、用户贡献优先。
